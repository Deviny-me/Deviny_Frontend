# Leaderboards API — спецификация для бэкенда

Документ описывает, что нужно от бэка, чтобы фронтовая страница `/.../leaderboards`
(а также сайдбар «Зал славы») получила полноценный поиск, сортировку, фильтрацию,
пагинацию, периоды и «моё место». Сейчас всё это сделано на клиенте по первым 100 записям —
этого недостаточно для масштабирования.

Базовый префикс предполагается `/api/ratings` (текущая структура `ratingsApi.ts`).
Если у вас другой — поправьте пути, остальное останется в силе.

---

## 1. `GET /api/ratings/leaderboard`

Расширение существующего эндпоинта получения рейтинга.

### Query-параметры

| Параметр       | Тип                                                | По умолч.                       | Описание                                                                 |
| -------------- | -------------------------------------------------- | ------------------------------- | ------------------------------------------------------------------------ |
| `category`     | `user` \| `trainer` \| `nutritionist`              | required                        | Какую категорию участников возвращать                                    |
| `scope`        | `global` \| `local`                                | `global`                        | См. п. «Семантика scope=local»                                           |
| `period`       | `week` \| `month` \| `season` \| `all`             | `week`                          | Окно расчёта рейтинга                                                    |
| `limit`        | int (1..100)                                       | 50                              | Размер страницы                                                          |
| `offset`       | int ≥ 0                                            | 0                               | Смещение для пагинации                                                   |
| `sort`         | `rank` \| `score` \| `stars` \| `reviews` \| `name`| `rank`                          | Поле сортировки                                                          |
| `direction`    | `asc` \| `desc`                                    | `desc` (для `name`/`rank` — `asc`) | Направление                                                            |
| `minReviews`   | int ≥ 0                                            | 0                               | Только записи с `ratingCount ≥ minReviews`                               |
| `q`            | string                                             | —                               | Поиск по `fullName` (case-insensitive, partial / `ILIKE %q%`)            |
| `country`      | string (ISO-3166-1 alpha-2)                        | —                               | Используется при `scope=local` (см. ниже)                                |
| `city`         | string                                             | —                               | Опционально, при `scope=local`                                           |

### Ответ

```json
{
  "items": [
    {
      "userId": "guid",
      "fullName": "string",
      "avatarUrl": "string|null",
      "role": "user|trainer|nutritionist",
      "starRating": 4.5,
      "overallScore": 87.3,
      "ratingCount": 128,
      "rank": 1
    }
  ],
  "total": 4321,
  "period": "week",
  "scope": "global",
  "generatedAt": "2026-05-10T12:34:56Z"
}
```

### Важные гарантии

- `rank` — это **глобальное место** в рамках выбранных `category + scope + period + minReviews`.
  Оно **не зависит** от `sort`, `q`, `offset`, `limit`. То есть если юзер реально 1-й по `rank`,
  он останется `rank: 1` даже когда ответ отсортирован по `name` или отфильтрован поиском.
  Это нужно фронту, чтобы золото/серебро/бронза всегда соответствовали настоящим местам.
- `total` — общее количество строк, удовлетворяющих фильтрам (без учёта `limit/offset`).
- `generatedAt` — UTC ISO-8601, для UI-индикатора «обновлено N минут назад» и для кэш-инвалидации.

### Tie-breakers (рекомендуемые)

- `sort=score`:   `overallScore desc → ratingCount desc → userId asc`
- `sort=stars`:   `starRating desc → ratingCount desc → overallScore desc`
- `sort=reviews`: `ratingCount desc → overallScore desc`
- `sort=name`:    `fullName ASC` (collation case-insensitive, locale-aware для ru/az)
- `sort=rank`:    стабильно по возрастанию `rank`

## 2. `GET /api/ratings/leaderboard/me`

Возвращает позицию текущего залогиненного пользователя.
Принимает те же фильтры, что и основной эндпоинт: `category`, `scope`, `period`, `minReviews`.

### Ответ

```json
{
  "userId": "guid",
  "fullName": "string",
  "avatarUrl": "string|null",
  "role": "user|trainer|nutritionist",
  "starRating": 4.2,
  "overallScore": 71.4,
  "ratingCount": 34,
  "rank": 412,
  "total": 4321,
  "percentile": 9.5
}
```

- Если юзер не подходит под `minReviews` или категорию — `rank: null`,
  `percentile: null`, остальные поля заполнены значениями профиля.
- Используется фронтом для пина «Твоё место — N из M» внизу страницы,
  даже когда юзер вне топ-100.

---

## 3. Поле `role`

В `LeaderboardEntryDto` сейчас `role?: string | null`. Просьба **всегда** возвращать одно из:
`user`, `trainer`, `nutritionist` (в нижнем регистре).
Фронт использует это поле для акцентного цвета строки.

---

## 4. Период (`period`)

- `week` — последние 7 дней (sliding) **или** текущая ISO-неделя — зафиксировать одно.
- `month` — последние 30 дней.
- `season` — текущий «сезон» рейтинга (если у нас есть концепт сезонов).
- `all` — за всё время.

`generatedAt` в ответе позволяет фронту корректно показывать «обновлено N минут назад».

---

## 5. Семантика `scope=local`

Сейчас на клиенте это просто переключатель без чёткой семантики. Нужно зафиксировать.
Предлагаемая семантика по умолчанию:

- `scope=local` = выборка ограничена `country` текущего пользователя
  (берётся из профиля caller-а).
- Если у пользователя `country` пустой — fallback на `global` + флаг в ответе
  (или 422 с понятным сообщением — по выбору).
- Опционально разрешить override через query: `?country=AZ` / `?city=Baku`
  (полезно для админ-инструментов).

---

## 6. Кэширование

OK кэшировать ответ на короткий TTL (30–60 сек) с ключом
`category + scope + period + minReviews + limit + offset + sort + direction + q + country + city`.
Желательны `ETag` / `Last-Modified` для условных запросов.

---

## 7. Ошибки

Стандартные:
- `400` — валидация query (неизвестный `sort`, отрицательный `offset`, и т.п.)
- `401` — для `/leaderboard/me`
- `429` — rate limit (если стоит)

Тело:

```json
{ "error": "human-readable", "code": "VALIDATION_ERROR" }
```

---

## 8. Приоритет реализации

Если делать поэтапно — порядок такой (фронт уже готов работать со всем этим,
включает фичи по мере появления параметров):

1. **Must-have (минимум, чтобы расширить функциональность сразу):**
   - `period` (хотя бы `week` и `all`)
   - `total` и `generatedAt` в ответе
   - гарантия, что `role` всегда заполнено

2. **Want (даст серверную сортировку/фильтрацию по всей базе, не только по топ-100):**
   - `q` (поиск)
   - `minReviews` (фильтр)
   - `sort` + `direction`

3. **Next (масштабирование за пределы топ-100 и «моё место»):**
   - `offset` + `limit` пагинация
   - `GET /leaderboard/me`

---

## 9. Текущее поведение фронта (для контекста)

Пока всё перечисленное не реализовано на сервере, страница работает так:

- Запрашивает `GET /leaderboard?category=...&scope=...&limit=100`.
- Поиск, сортировка (`rank` / `score` / `stars` / `reviews` / `name`)
  и фильтр по числу отзывов (`all` / `rated` / `min10` / `min50`)
  применяются **на клиенте** в пределах загруженных 100 строк.
- Медали 1/2/3 берутся по полю `rank` из ответа и не «съезжают» при ресорте/фильтре.
- Строка «Топ-10 на этой неделе» в шапке сейчас декоративна — фактический период
  сервером не учитывается.

После появления параметров из раздела 8 (Want / Next) клиентская фильтрация
будет постепенно заменена на серверную, и список перестанет быть ограничен топ-100.
